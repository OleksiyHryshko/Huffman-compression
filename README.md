# Huffman Compression (C++)

Small command-line file compressor/decompressor written in C++ that implements **Huffman coding**.

This was a **first-semester programming** lab project. I’m keeping it public as a learning artifact: the core algorithm works, the code is readable, and there are also a few limitations that I call out below (plus ideas for improvements).

## What it does

Given an input file, the program:

1. Counts byte frequencies.
2. Builds a Huffman tree using a min-priority queue.
3. Generates a prefix-free bit code for each byte.
4. Writes:
   - a **compressed binary output** file
   - a **dictionary** file that maps `byte -> bitstring`

It also supports decompression using the dictionary.

## Files

- `ConsoleApplication1.cpp` — `main()` + mode selection (`k` compress / `d` decompress)
- `Header.h`, `Header.cpp` — implementation (Huffman tree build, encode, I/O helpers)
- `ConsoleApplication1.sln`, `.vcxproj` — Visual Studio project files (Windows)

## Usage

The program accepts:

```text
-i <input file>
-o <output file>
-s <dictionary file>
-t <mode: k|d>

Modes:
  k - compression
  d - decompression
```

### Compress

```bash
./huffman -i input.txt -o compressed.bin -s dict.txt -t k
```

### Decompress

```bash
./huffman -i compressed.bin -o restored.txt -s dict.txt -t d
```

## Dictionary format

The dictionary is stored as text lines:

```text
<symbol><bitstring>\n
```

Example (illustrative):

```text
a101
b00
 111
```

Notes:

- The first character of each line is the symbol (a raw byte/character).
- The rest of the line is its Huffman code as `0`/`1`.
- Newline `\n` is handled with a special case in `Readfromdictionary()`.

## Implementation notes (high level)

- Huffman tree is built with a `std::priority_queue` (min-heap via custom comparator).
- Codes are generated by recursive traversal:
  - left edge appends `0`
  - right edge appends `1`
- Compressed output is written as bytes by taking the bitstring and packing it into 8-bit chunks.

## Known limitations (honest)

This is a student project, so there are some important limitations that I’d fix in a production version:

1. **Bit padding is not stored**
   - Compression packs bits into bytes in chunks of 8.
   - If the total bit count isn’t a multiple of 8, the last chunk is ambiguous during decompression.
   - A typical fix is to store the number of valid bits in the last byte (or store original size).

2. **Decompression is O(n · m)**
   - Decoding currently scans the entire dictionary for each growing bit prefix:
	 `for (code in dictionary) if (code == prefix) ...`
   - Faster options: build a decoding trie/tree from codes, or use an inverse map and incremental prefix matching.

3. **Argument parsing bug**
   - In `GETPARAMS()`, the check `if (mode == 'k' || 'd')` is always true in C++.
   - (This doesn’t break typical use, but it’s logically incorrect.)
